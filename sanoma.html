A design system from scratch for Sanoma Pro

Breaking down a titan

Sanoma’s Kampus e-learning platform had grown too fast, and the product team was experiencing a communication breakdown. Designs were very out of date and didn’t match the developed product in many key aspects. There was no single source of truth for anyone to refer to - so we built one.

My tasks
  • Audit the current product and find inconsistencies
  • Redefine all base styles and primitives
  • Redesign every single component in the product
  • Template key screens and workflows to speed up design
  • Define ways of working for designers
  • Figma trainings for other designers
  • Help developers set up Angular Storybook

Design systems are a hassle to build. They need time, expertise, patient collaboration and lots of trial and error. All of that translates to monetary costs. However, if you think it's just too costly to bother with, wait till you see how much it costs not to have one.

Here are just some of the problems at Kampus that could have been solved with a proper design system implemented before their growth.

  • No source of truth to check designs and code against
  • No place to test components and their states before deploying to production
  • Unused legacy styles and code cluttering up design files and codebase
  • Designers and developers having trouble understanding and using past work
  • Hastily implemented designs with mistakes and accessibility errors
  • Developers reinventing the wheel repeatedly, with subtle differences each time
  • Pointless alignment meetings and answering same questions repeatedly
  • No standard format for documentation
  • Missing or inconsistent interaction states
  • Different parts of the product can't be reused on other pages due to legacy code or different implementation

As you might have guessed already, a lot of these problems above can be 'invisible'. Since they're happening constantly in the background, it's hard to even realise they're a nuisance that can be dealt with. So we did an audit to make the invisible painfully visible. In a nutshell, we went through the UI (designs and coded product) with a fine tooth comb.

  • 30+ unused or redundant color styles
  • Dozens of inaccessible or conflicting type styles
  • 55+ components with outdated or broken design
  • Wildly varying ways of working in Figma and code
  • Almost every component had missing interaction states
  • Limited responsiveness guidelines and no grid system
  • Inefficient use of Figma features like auto layout and constraints
  • Non responsive designs → confusion when implementing
  • Inconsistent file hygiene practices (layer naming, context grouping, grouping, user flow formatting)
  • No standard format for documentation

Exploration and iteration

Initially I explored the many different ways a user would be able to create an annotation: from the sidebar, toolbar, header, comment input, and file preview. I also explored different annotation types: area annotations, point annotations, drawing and free-form, and highlights. As part of my process, I detailed out every single flow and weighed the pros and cons of each.

Although the most intuitive entrypoint was from the file preview itself, this was technically not feasible in the time span that we had. I resorted to two different entrypoints: the file toolbar and the comment input in the sidebar. We also settled on three different annotation types (point, drawing, and highlights) which solved for our primary use cases: marking up contracts and other documents like homework, and giving location-specific feedback on images in creative domains.
